Interview questions — Summary table

Looking only at the table, which frameworks are credible for a Microsoft-shop production deployment in 2026?

MAF first (.NET parity, Foundry hosting, durable workflows, native HITL, OTel-native, GA 1.0). Semantic Kernel as a still-supported alternative if already in production. AutoGen for research/prototyping. LangGraph as the strongest non-Microsoft cross-cutting alternative.

Which frameworks combine durable workflow checkpointing AND first-class HITL?

Only MAF and LangGraph (and SK Process Framework as a state-machine variant). Everything else either has weak persistence or DIY HITL.

Which frameworks are language-portable beyond Python?

MAF (Python + .NET), Semantic Kernel (.NET + Python + Java), LangGraph (Python + JS/TS), OpenAI Agents SDK (Python + TS), LlamaIndex (Python + TS), Letta (Python + TS), Google ADK (Python + Java).

Which frameworks ship a managed runtime in addition to the OSS library?

MAF (Foundry-hosted), LangGraph (LangGraph Platform), CrewAI (Crew Enterprise), LlamaIndex (Llama Cloud), Haystack (deepset Cloud), Letta (Letta Cloud), AutoGPT (AutoGPT Platform), Google ADK (Vertex Agent Engine). OpenAI Agents SDK and Pydantic AI deliberately ship library-only.

If license matters (Apache 2.0 over MIT), what are your options?

Haystack, Smolagents, Letta, Google ADK. Everything else here is MIT (or Polyform Shield in the case of AutoGPT Platform).

2. Detailed framework-by-framework notes

2.1 Microsoft AutoGen #

Repo microsoft/autogen  ·  Docs microsoft.github.io/autogen  ·  License MIT

Problem. AutoGen pioneered the conversable multi-agent pattern: arbitrary teams of LLM-driven agents (and humans) collaborate by messaging each other. It originated as a Microsoft Research project to study how LLM agents could plan, code, critique and self-correct in groups.

When to use. Research on multi-agent collaboration; debate / planner-executor patterns; prototyping novel orchestration (Magentic-One, Swarm, GroupChatManager) before standardising on a production platform; code-heavy agentic tasks where the CodeExecutor (Docker / Jupyter) is a first-class citizen.

Architecture (since v0.4, Jan 2025). Three-layer:

  1. autogen-core — async, event-driven actor runtime. Single-process or distributed (gRPC).
  2. autogen-agentchat — high-level conversational API: AssistantAgent, UserProxyAgent, RoundRobinGroupChat, SelectorGroupChat, Swarm, MagenticOneGroupChat.
  3. autogen-ext — extensions: model clients, code executors (Docker, Jupyter, ACI), tools, web surfers, MCP.

Plus AutoGen Studio (low-code UI on AgentChat) and AutoGen Bench for evaluation.

Lifecycle. Build → register with runtime → start runtime → publish task → agents subscribe and respond → termination condition → runtime stop. Streaming via async iterators of BaseChatMessage.

Tool-calling. FunctionTool wraps Python callables. MCP tools first-class via mcp_server_tools(). OpenAI-style tool calls; results fed back as ToolCallResultMessage.

Memory. Stateless by default; AssistantAgent keeps a model context (sliding window, buffered, head-and-tail) plus a pluggable Memory provider (Chroma, Mem0, custom). Long-term memory is intentionally an extension point.

Multi-agent orchestration. Headline feature.

  • RoundRobinGroupChat — deterministic rotation.
  • SelectorGroupChat — an LLM picks the next speaker.
  • Swarm — agents handoff via tool calls.
  • MagenticOneGroupChat — orchestrator coordinates web/file/code/coder agents.
  • Custom: subclass BaseGroupChat or wire Core actors directly.

Workflow. Treats orchestration as conversation, not as a DAG — no first-class graph workflow type (the gap MAF closes).

HITL. UserProxyAgent participates as human; input_func wires CLI/web/Slack inputs.

Eval / Observability. autogen-bench (HumanEval, GAIA, AssistantBench). OpenTelemetry traces and metrics across runtime + AgentChat. AutoGen Studio session viewer.

Deployment. Local Python; distributed runtime over gRPC; Studio is FastAPI. Containerisable but not opinionated.

Enterprise. Mid-tier. OTel + sandboxed code-execution good; durable workflow checkpointing and managed hosting are not first-class — the exact gap MAF was created to close. Microsoft's official statement: AutoGen continues as a research playground; production users are guided to MAF.

Pros. Best-in-class multi-agent conversation primitives; rich research patterns; strong code-execution; healthy community.
Cons. API churn (0.2 → 0.4); no native durable workflow / time-travel; production deployment is DIY; .NET community-only.

vs MAF. AutoGen is the research lineage; MAF is the production lineage with same agent ergonomics plus typed graph workflows, checkpointing, hosting, and .NET parity.

Interview questions — AutoGen

Walk me through AutoGen v0.4's three layers and what each is for.

Core (event-driven actor runtime, async messaging, single-process or distributed gRPC), AgentChat (conversational high-level API: AssistantAgent / UserProxyAgent / RoundRobin / Selector / Swarm / Magentic), Extensions (model clients, code executors, tools, memory). Studio + Bench sit above AgentChat.

When do you pick SelectorGroupChat vs Swarm vs MagenticOneGroupChat?

Selector when an LLM should pick the next speaker in an open conversation; Swarm when specialist agents own the conversation and explicitly hand off; Magentic when you need a planner/orchestrator with a task ledger and progress tracking for long open-ended goals.

How does AutoGen handle code execution safely?

Through pluggable CodeExecutors — DockerCommandLineCodeExecutor, JupyterCodeExecutor, AzureContainerCodeExecutor. Each runs in an isolated container/kernel with no host filesystem access; egress can be locked down; results stream back as CodeResult messages.

A team says "we love AutoGen but need production durability and HITL." What do you recommend?

Migrate to MAF: same agent feel plus typed graph workflows, durable checkpointer, time-travel, RequestInfoExecutor HITL, .NET parity, OTel-native, and Foundry-hosted runtime. Use Microsoft's official migration guide; run both side-by-side and diff traces.

How do you prevent runaway loops in an AutoGen group chat?

Use MaxMessageTermination, TextMentionTermination, structured "DONE" flags, per-thread token/cost budgets, and Magentic-style progress ledgers that detect no-progress and either replan or escalate to UserProxyAgent.

How is UserProxyAgent actually used for HITL?

Treat it as a peer agent that, instead of calling an LLM, calls an input_func (CLI / web / Slack). It can also auto-execute code blocks. Termination conditions can target the human's reply.

2.2 Microsoft Agent Framework (MAF) #

Repo microsoft/agent-framework  ·  Docs learn.microsoft.com/agent-framework  ·  License MIT

Problem. Unify and productionise the best ideas from AutoGen and Semantic Kernel into a single, multi-language, opinionated framework: ergonomic agents, graph workflows, durable state, observability, governance, and turnkey hosting on Microsoft Foundry — while staying provider-agnostic.

Architecture.

  • AIAgent — provider-agnostic chat agent (OpenAIChatClient, AzureOpenAIChatClient, FoundryChatClient, AnthropicChatClient, BedrockChatClient, OllamaChatClient). Backed by Microsoft.Extensions.AI (IChatClient).
  • Tools — function tools, MCP, OpenAPI, Foundry-hosted (Bing, file search, code interpreter).
  • Middleware — request/response/exception pipelines (telemetry, retries, redaction, security).
  • Workflows — typed graph engine (Workflow, Executor, Edge, WorkflowBuilder): sequential, concurrent, handoff, group chat, Magentic, custom; streaming events; checkpointing, resume, time-travel, sub-workflows.
  • Memory — IMemory providers (Foundry, Redis, custom); thread-scoped AgentThread.
  • Skills — domain knowledge bundles.
  • Hosting — local; ASP.NET Core; FastAPI; Foundry-hosted agents (deploy with two extra lines); Agent Harness for shell/file/messaging loops.
  • DevUI — browser debugger that visualises message flow, tool calls, state, and workflow graphs.
  • Adapters — AG-UI, CopilotKit, ChatKit.

Lifecycle. Create chat client → build AIAgent (instructions, tools, memory) → optional middleware → invoke (streaming or not) → AgentThread carries state → optional checkpoint → optional rehydration / handoff to workflow.

HITL. First-class RequestInfoExecutor pauses workflow and emits a RequestInfo event; host returns RespondInfo to resume. Tool-call approval flows; UI adapters.

Observability. Native OpenTelemetry — spans for agent run, model call, tool call, executor, workflow event. Drop-in for App Insights, Jaeger, Honeycomb, Langfuse, Phoenix. DevUI consumes the same OTel feed locally.

Eval. Preview workflow-level evaluators (regression, golden-set, LLM-as-judge), agent-framework-lab, Foundry Evaluations integration.

Maturity. GA 1.0 since April 2026 with LTS commitment for core building blocks (agents, workflows, memory, middleware, orchestration). DevUI, evaluations, Foundry-hosted agents and front-end adapters remain in preview.

Pros. True .NET + Python parity; graph workflows with durability/checkpointing/time-travel/HITL; provider-agnostic; DevUI; clean OTel.
Cons. Younger community; some preview features still moving; steeper than CrewAI for newcomers.

Interview questions — Microsoft Agent Framework

What are MAF's core building blocks?

AIAgent, tools (functions / MCP / OpenAPI / Foundry-hosted), middleware, AgentThread, IMemory, Skills, Workflows (Workflow/Executor/Edge/WorkflowBuilder), hosting surfaces, DevUI, and front-end adapters.

What is RequestInfoExecutor and why is it important?

A workflow node that pauses execution, persists a checkpoint, and emits a typed RequestInfo event to the host. The host (UI, Slack, ticketing system) returns a RespondInfo that resumes the workflow at the same checkpoint. This is how MAF gets first-class HITL without bolting it onto a chat loop.

What's the difference between an AIAgent and a workflow Executor?

An AIAgent is an LLM-driven actor with tools, memory and instructions. An Executor is a typed node in a graph workflow — it can wrap an agent, a deterministic function, a HITL pause, or a sub-workflow. Workflows compose executors; agents are one kind of executor.

Walk me through deploying to Foundry-hosted agents.

Build the agent or workflow locally; add .WithFoundryHosting(...) (.NET) or the Python equivalent; configure your project endpoint and identity; the framework pushes the agent definition to Foundry, which provisions managed compute, autoscale, networking, identity (Entra), telemetry (App Insights) and policy controls.

How does MAF stay provider-agnostic?

Through Microsoft.Extensions.AI.IChatClient (and the Python protocol equivalent). Concrete implementations exist for Foundry, Azure OpenAI, OpenAI, Anthropic, Bedrock, Ollama, GitHub Copilot SDK and Claude Code SDK; swapping providers is a configuration change.

Which MAF features are GA in 1.0 vs preview?

GA / LTS: agents, workflows, memory, middleware, orchestration patterns, checkpointing, OTel observability, declarative YAML agents.
Preview: DevUI, Foundry-hosted agents, Foundry-backed memory and tools, evaluations, AG-UI / CopilotKit / ChatKit adapters, reusable Skills, Copilot/Claude Code SDK integration, Agent Harness.

How would you migrate an AutoGen 0.4 group chat to MAF?

Map AssistantAgent → AIAgent; replace SelectorGroupChat with the MAF group-chat workflow pattern (or a typed handoff graph); replace UserProxyAgent with RequestInfoExecutor plus a CopilotKit/AG-UI adapter; map memory provider to IMemory; turn on OTel + DevUI to verify behavioural parity; add a checkpointer for durability.

How does MAF observability work end-to-end?

Native OpenTelemetry spans for agent.run, llm.call, tool.call, executor, workflow.event, request_info. Attributes follow the GenAI semantic conventions (model, tokens, cost, tool name, redacted args, thread_id). Spans flow to App Insights / Jaeger / Honeycomb / Phoenix / Langfuse / DevUI.

2.3 Semantic Kernel #

Repo microsoft/semantic-kernel  ·  Docs learn.microsoft.com/semantic-kernel

Problem. Microsoft's first-generation "AI orchestration SDK": LLM functions as plugins inside a Kernel, planners and chat-completion agents that fit into existing C# / Python / Java apps.

Highlights. Plugins / connectors / filters; Process Framework (state-machine workflows); Agent Framework in SK is the lineage that became MAF.

Status. Mature; in convergence mode. SK as a Kernel/plugin SDK remains supported; agent-specific APIs are guided to MAF. New projects should start on MAF unless tied to SK Process Framework.

Interview questions — Semantic Kernel

What problem did Semantic Kernel originally solve?

A polyglot (.NET / Python / Java) DI-friendly orchestration SDK that treats LLM functions as plugins inside a Kernel, plus connectors and planners — designed to drop into existing enterprise apps rather than imposing a new runtime.

What is the SK Process Framework and when use it over a MAF Workflow?

Process Framework models long-running business processes as state machines with steps, events, parallel branches and retries. It predates MAF Workflows and remains the right choice when you already have SK in production and your model is process state machine rather than agent graph. New projects with HITL + durable agent graphs should prefer MAF Workflows.

How are SK plugins different from MAF tools?

Plugins are collections of KernelFunctions (native code or prompt templates) registered with the Kernel — first-class KernelArguments, prompt-template functions, and DI integration. MAF tools are AIFunction/Python @ai_function callables registered with an agent; simpler surface, but tools can also be MCP, OpenAPI, or Foundry-hosted.

How are SK filters different from MAF middleware?

SK filters are sync/async hooks around prompt rendering and function invocation — good for redaction, telemetry, retries inside a Kernel call. MAF middleware sits around the agent run and tool call lifecycle and is composable across the workflow. Same intent, different scope.

When should a team migrate from SK Agent Framework to MAF?

When they need durable graph workflows, time-travel, native HITL primitives, .NET + Python parity for agents, or Foundry-hosted runtime. SK as a Kernel/plugin SDK can still coexist.

2.4 LangGraph #

Repo langchain-ai/langgraph  ·  Docs docs.langchain.com/oss/python/langgraph

Problem. Bring graph + state-machine discipline to LLM control flow: explicit nodes, edges, reducers, persistence and interrupts.

Architecture. StateGraph with TypedDict state and reducers; nodes are sync/async Python functions; conditional edges; pluggable Checkpointer (MemorySaver, SqliteSaver, PostgresSaver, AWS Bedrock Sessions); interrupt() + Command(resume=…) for HITL; LangSmith for tracing/eval; LangGraph Studio for visualisation; LangGraph Platform for managed runtime.

Multi-agent. langgraph-supervisor, langgraph-swarm; hierarchical and network patterns.

Pros. Graph + persistence is the right abstraction; HITL and time-travel are native; LangSmith.
Cons. Best inside LangChain ecosystem; reducer schema feels heavy at first.

vs MAF. Closest peer. MAF wins on .NET parity, Foundry hosting, declarative agents and Microsoft ecosystem; LangGraph wins on community size and LangSmith.

Interview questions — LangGraph

Explain reducers in StateGraph.

State is a TypedDict whose fields are Annotated[T, reducer]. After each node returns a partial update, the runtime applies the reducer to merge it into the current state (e.g., add for list concatenation, operator.or_ for set union, custom for typed merges). Reducers prevent overwrite races and let multiple nodes update the same field safely.

How does interrupt() actually work?

The runtime raises a special exception carrying a payload; the graph's checkpoint at that step is persisted. The host catches the value, asks the human, then resumes via graph.invoke(..., Command(resume=value)) which seeks to the checkpoint and continues with the human input merged into state.

MemorySaver vs SqliteSaver vs PostgresSaver vs BaseStore — when each?

MemorySaver for tests and local dev. SqliteSaver for single-process apps with simple persistence. PostgresSaver for production with multiple workers and horizontal scale. BaseStore (separate from checkpointers) for cross-thread long-term memory (user preferences, KB).

Supervisor vs Swarm packages — which when?

langgraph-supervisor for one-coordinator-many-workers (research writer, support orchestrator). langgraph-swarm for peer-to-peer handoffs where the "right" specialist owns the conversation (multi-domain triage).

How does LangSmith integrate with LangGraph?

The LangChain runtime auto-emits traces to LangSmith including the per-step state delta, model call, and tool call; you can attach datasets, run regression evals, and replay any trace. LangSmith's UI is graph-aware so you can see the same nodes you built.

When should you reach for Temporal instead of LangGraph?

When workflows can run for days/weeks with cross-system side effects, exactly-once guarantees, complex retry/backoff matrices, and you need the operations team to manage the runtime. Pattern: keep agent reasoning inside LangGraph; orchestrate the surrounding business workflow in Temporal.

How is LangGraph different from old-style LangChain AgentExecutor?

AgentExecutor was an opaque ReAct loop with no first-class state, persistence, or HITL. LangGraph exposes the loop as an explicit graph you control, persists every step, supports interrupts, and is the recommended replacement.

2.5 CrewAI #

Repo crewAIInc/crewAI  ·  Docs docs.crewai.com

Problem. Multi-agent collaboration that feels like staffing a team: Roles, Goals, Tasks and Crews, plus Flows for event-driven production pipelines.

Lifecycle. Define roles → instantiate crew → crew.kickoff(inputs) → tasks run sequentially or via manager → outputs flow → optional Flow handles broader pipeline.

Pros. Lowest friction for role-based multi-agent; great for business teams; Flows give production-ish layer.
Cons. Less control over LLM cost/latency; emergent failure modes.

Interview questions — CrewAI

Crews vs Flows — when use which?

Crews for autonomous, role-based collaboration where agents decide who delegates and how (research, content, marketing). Flows for production pipelines that need event-driven, deterministic structure with state, conditional logic and triggers. Most real systems compose them: a Flow orchestrates one or more Crews.

How does the hierarchical Process work?

A manager LLM is given the list of agents and tasks; for each task it picks the best-suited agent, dispatches, observes output, and either accepts or routes for revision — like an LLM-driven foreman.

What memory layers does CrewAI provide?

Short-term (current run), long-term (across runs, default mem0-backed), entity memory (per-entity facts), and a tool cache (deterministic tool re-runs). Per-crew and per-agent.

How would you scale CrewAI to many concurrent users?

Run each kickoff in a stateless worker (FastAPI / Celery / Cloud Run); externalise memory (mem0 / Redis); cap tools and tokens per task; add Langfuse/OpenLIT tracing; for production guarantees consider wrapping the Flow in Temporal or moving the orchestration to LangGraph/MAF.

What are CrewAI's limitations to be aware of?

Loose reasoning traces, less control over LLM cost/latency than graph frameworks, hierarchical process can over-rely on the manager LLM, and durability/checkpointing is partial. Mitigate with budget guardrails and explicit Flows.

2.6 OpenAI Agents SDK #

Repo openai-agents-python / openai-agents-js

Architecture. Pure Python (and TS): Agent(name, instructions, tools, handoffs, guardrails); Runner loop drives turns; Sessions hold thread state; Handoffs are tools the LLM emits; Guardrails run async checks in parallel; tracing exports to OpenAI dashboard or any OTel/Logfire/Langfuse exporter.

Pros. Minimal, ergonomic, great handoffs/guardrails/tracing; provider-agnostic; TS parity.
Cons. No graph workflow; no durable runtime; hosted tracing is OpenAI by default.

Interview questions — OpenAI Agents SDK

How are handoffs implemented under the hood?

Each handoff is exposed to the LLM as a tool whose schema represents "transfer control to agent X". When the model calls it, the runner switches the active agent and feeds the conversation forward; optional input_filters strip or transform context before the next agent sees it.

What is a guardrail and when does it run?

An async function that runs in parallel with the agent step. It returns tripwire=True to halt execution. Input guardrails run on user input; output guardrails run on agent output; tool guardrails run around tool calls. They fail-fast and never block the happy path.

How does Sessions work?

Sessions persist conversation state (thread of messages, tool results) outside the runner, keyed by an ID. Restart the process or move to another worker — passing the same session resumes the conversation.

Compare agents-as-tools with handoffs.

Agents-as-tools: parent retains control, calls a child agent like a function and synthesises the reply. Handoffs: control transfers entirely. Pick agents-as-tools for orchestrator/specialist; pick handoffs for triage/routing.

What does the tracing dashboard show?

Per-run: spans for each model call, tool call, handoff, and guardrail; inputs/outputs, latency, and costs. Pluggable trace processors export the same spans to Logfire / Langfuse / Phoenix / Datadog or any OTel backend.

What's missing from OpenAI Agents SDK that LangGraph/MAF provide?

A graph workflow engine, durable checkpointing, time-travel, typed multi-step state, and a managed runtime — by design. For complex flows, embed the SDK inside Temporal/LangGraph/MAF.

2.7 LlamaIndex Workflows / AgentWorkflow #

Repo run-llama/llama_index

Architecture. Event-driven Workflow with @step handlers reacting to events on a shared Context. AgentWorkflow orchestrates FunctionAgent / ReActAgent with handoffs.

Pros. Best RAG integration of any agent framework; clean event semantics.
Cons. Smaller multi-agent pattern catalogue; primarily Python.

Interview questions — LlamaIndex AgentWorkflow

How is LlamaIndex Workflow different from LangGraph StateGraph?

Workflow is event-driven: @step handlers consume events and emit new events; the framework dispatches by event type. StateGraph is node/edge with reducers over a typed state. Both are graph engines; events feel more natural for streaming and pub/sub-style flows, state graphs are better when "the state" is the central concept.

What is Context and what does it buy you?

A shared kv store + streaming sink across all steps in a workflow run. Steps read/write context for cross-step data without re-emitting events; tools also access it. It's the LlamaIndex equivalent of LangGraph's state.

How does AgentWorkflow do handoffs?

It takes a list of agents (with declared tools and handoffs); the orchestrator agent emits a handoff event; the workflow routes it to the named agent which then becomes the active agent for subsequent events. Context is preserved.

FunctionAgent vs ReActAgent — which when?

FunctionAgent for native function-calling models (OpenAI/Anthropic) — single tool-call message per step. ReActAgent for non-function-calling models or when you want the thought / action / observation trace explicit (text-based reasoning).

Why is LlamaIndex AgentWorkflow particularly good for RAG-heavy agents?

Tools include QueryEngineTool, RetrieverTool, and seamless integration with the LlamaIndex retrieval stack (sub-question, hierarchical, hybrid). Agents call retrieval as just another tool, but the tool can carry rich citations and structured data back through Context.

2.8 Haystack Agents #

Repo deepset-ai/haystack  ·  License Apache 2.0

Architecture. Typed Components assembled into Pipelines (DAG with branches and loops). Agent is a pipeline component looping a ChatGenerator with a ToolInvoker.

Pros. Type-safe pipelines; great retrieval; Apache 2.0; deepset Studio.
Cons. Multi-agent abstractions thinner; less "agentic" mindshare.

Interview questions — Haystack

What's the difference between Tool, ComponentTool, and the @tool decorator?

Tool is the base abstraction (name, description, JSON schema, callable). ComponentTool wraps any Haystack component as a tool — its inputs/outputs become the schema automatically. @tool decorates a Python function and uses its signature + docstring. Pick @tool for ad-hoc functions, ComponentTool to expose existing pipelines as tools.

How does the Agent component loop work?

It feeds the system prompt + history to a ChatGenerator; if the response includes tool calls, it invokes them via a ToolInvoker, appends results to history, and re-invokes the generator until the model returns a non-tool message or termination. State is injected through the live State parameter.

Why pick Haystack for an Apache 2.0 stack?

Most agent frameworks are MIT, but some enterprises require Apache 2.0 for patent grant clarity. Haystack is permissive Apache 2.0 with corporate backing (deepset) and a strong retrieval pedigree.

How do branches and loops work in pipelines?

Components declare typed inputs/outputs; the pipeline DAG can fan out (multiple components consume the same output) and loop (an Agent can call itself or an upstream component again until a condition holds). Loops require explicit termination guards.

When pick Haystack over LangGraph for a RAG agent?

When you want type-safe pipelines, strong retrieval primitives, and don't need a deep multi-agent or HITL story. For pure agent + workflow + HITL, LangGraph or MAF is stronger.

2.9 AutoGPT / Forge #

The original "give a goal, get an autonomous agent" project pivoted to:

  • AutoGPT Platform — low-code visual block-graph builder + marketplace + credits + multi-tenant infra. License: Polyform Shield (no commercial redistribution).
  • AutoGPT Classic + Forge — original autonomous agent + SDK template. MIT.

Now closer to a product than an embeddable SDK; useful as a low-code platform but not a typical embedded library choice.

Interview questions — AutoGPT / Forge

What's the relationship between AutoGPT Platform, AutoGPT Classic, and Forge?

AutoGPT Classic is the original autonomous-agent project. Forge is an SDK template for building Auto-GPT-style agents (abilities, memory, LLM abstraction). AutoGPT Platform is a newer block-graph low-code product with marketplace and credits — a different category from the original library.

Why does the Polyform Shield license matter?

It restricts commercial redistribution of AutoGPT Platform — fine for internal business use, problematic if you want to embed it in a commercial product or fork it for resale. Forge and Classic remain MIT.

Why has AutoGPT lost developer mindshare to LangGraph and MAF?

Production teams need typed graphs, durability, HITL, OTel, and managed runtime; AutoGPT's sweet spot moved to a product ("low-code visual builder") rather than a library. The original autonomous-loop pattern was inspirational but not directly productionable.

When (if ever) would you pick AutoGPT today?

When you want a low-code visual builder with a marketplace and a community of pre-built blocks for non-developers, and the Polyform license is acceptable. For embedded library use, prefer MAF / LangGraph / OpenAI Agents SDK.

2.10 Other notable frameworks #

Pydantic AI

"FastAPI feel" agents: type-safe Agent[deps, output], structured outputs via Pydantic, A2A and MCP, DI, Logfire observability.

Smolagents (HF)

Code-writing agents — LLM writes Python instead of JSON tool calls. CodeAgent + ToolCallingAgent + ManagedAgent.

Letta (ex-MemGPT)

Stateful agents with hierarchical memory (core / archival / recall) running as a server. Good for long-term memory across sessions.

MetaGPT

"Software company" of role-based agents (PM, architect, engineer, QA) producing PRDs, designs and code via SOP-driven hierarchy.

Google ADK

Python + Java SDK: LlmAgent, SequentialAgent, ParallelAgent, LoopAgent, sessions, callbacks, A2A, Vertex AI integration.

Mastra (TS)

TypeScript-first agent framework; nice for Node-only shops.

Interview questions — Other frameworks

Why pick Pydantic AI?

When the team values type-safety, structured outputs (Pydantic models for both inputs and outputs), DI, and a "FastAPI for agents" feel — and a single-agent or light-multi-agent shape is sufficient. A2A and MCP are first-class; Logfire gives clean observability.

What's the trade-off with Smolagents' code-as-action approach?

Strong models produce solid Python; weaker models (sub-7B) introduce subtle bugs. Sandboxing is mandatory. The upside is composability — the agent can express loops, conditionals, and data manipulation as code instead of repeated tool-call rounds.

When does Letta beat building memory yourself?

When you need hierarchical, server-managed long-term memory across sessions and devices and you don't want to design eviction, archival, recall and consistency. Embed Letta as a memory backend behind a MAF or LangGraph agent.

What's MetaGPT's niche?

Generating end-to-end software artifacts (PRD, architecture, code, tests) from a single brief, by simulating a software company with role-based agents. Less flexible for general agent workflows but a reference for SOP-driven hierarchies.

How does Google ADK compare to MAF?

Strong on Google Cloud / Gemini side with LlmAgent, SequentialAgent, ParallelAgent, LoopAgent, sessions, callbacks, and Vertex AI integration. MAF wins on .NET parity and the broader provider matrix; both speak A2A and MCP for cross-vendor interop.

Why is the legacy LangChain AgentExecutor deprecated?

It hides state, has no first-class HITL, no checkpointing, and a confusing failure model. LangGraph replaces it with explicit graphs, persistence, and interrupts.

7.1 Conceptual #

What distinguishes an "agent" from a "chatbot" or a "RAG pipeline"?

An agent autonomously plans actions, picks tools, observes outcomes, and iterates toward a goal — i.e. it has a control loop that mutates state. A chatbot is a single-turn (or multi-turn) responder with no autonomous tool use. A RAG pipeline is a static retrieval + generation flow with no decision-making about what to do next. The defining axis is autonomy over the action space: which tool to call, when to stop, when to ask a human, when to delegate.

Compare ReAct, Plan-and-Execute, ToT and Reflexion. When would you choose each?

ReAct interleaves thought → action → observation in a tight loop. Cheap and effective; default for most agents.
Plan-and-Execute first builds a multi-step plan, then runs it. Better for tasks where the cost of exploration is high or steps must be parallelisable.
Tree-of-Thoughts (ToT) explores branching reasoning trees with a value function. Use for hard combinatorial search (proofs, puzzles); expensive.
Reflexion adds a self-critique pass that updates an episodic memory after each attempt. Useful when retries are cheap and consistency matters (code, evals).

Why did Microsoft create Agent Framework when AutoGen and Semantic Kernel already existed?

AutoGen lacked production-grade durability (no first-class graph workflows, no checkpointing, weak hosting story). Semantic Kernel had enterprise discipline but its agent abstractions were lighter than AutoGen's. MAF unifies both lineages into one framework with shared concepts in Python and .NET, adds typed graph workflows, durable state with time-travel, native HITL, OpenTelemetry, declarative YAML agents, DevUI, and Foundry-hosted runtime — the gaps that prevented teams from going to production on either predecessor.

What is MCP and why does it matter?

The Model Context Protocol is an open standard for exposing tools, prompts and resources to LLMs over a structured RPC channel (stdio, SSE, HTTP). It matters because it decouples tool implementation from agent runtime: a single MCP server (e.g. for GitHub or a SQL database) can be consumed by AutoGen, MAF, LangGraph, Claude Code, Cursor, OpenAI Agents SDK, etc. without re-implementing tool wrappers per framework. It is the "USB-C of agent tools".

Explain "agents-as-tools" vs "handoffs". When is each appropriate?

Agents-as-tools exposes a sub-agent as a callable tool — the parent agent retains control and the sub-agent's reply is just another tool result. Best for orchestrator/specialist patterns where the orchestrator must decide and synthesise.

Handoffs transfer control entirely to another agent, which then drives the conversation (and may handoff again). Best for triage / routing patterns (customer support → billing specialist).

OpenAI Agents SDK and AutoGen Swarm use handoffs. MAF supports both via Workflow handoff edges and tool-style sub-agents.

7.2 Architecture & design #

Walk me through the layered architecture of AutoGen v0.4.

Three layers. Core is an asynchronous, event-driven actor runtime (single-process or distributed via gRPC) that defines an AgentRuntime, addressable agents, async messages, topics and subscriptions. AgentChat is a high-level API on top: AssistantAgent, UserProxyAgent, group chat patterns (RoundRobin, Selector, Swarm, MagenticOne) and termination conditions. Extensions provide model clients, code executors (Docker/Jupyter/ACI), tools and memory providers. AutoGen Studio and Bench sit above AgentChat for low-code authoring and benchmarking.

Walk me through the MAF Workflow engine and how it differs from AutoGen group chat.

A MAF Workflow is a typed directed graph of Executor nodes connected by Edges; messages flowing along edges are typed contracts. Patterns (sequential, concurrent fan-out/fan-in, handoff, group chat, Magentic) are pre-built Workflow templates. The engine streams events to consumers, persists state via a checkpointer after each step, and supports resume and time-travel (jump to a previous checkpoint and re-execute). HITL is modelled as a RequestInfoExecutor that pauses and emits a RequestInfo event the host fulfils.

AutoGen group chat is a conversation abstraction: agents pass natural-language messages and a manager picks who speaks. There is no typed graph, no built-in checkpointer, and HITL is a UserProxyAgent participant. MAF's typed graph wins on durability, governance, and predictable structure; AutoGen's conversation wins on flexibility and emergent reasoning.

How does LangGraph's StateGraph + reducers actually work?

You declare state as a TypedDict where each field is annotated with a reducer (e.g. Annotated[list, add]). A graph is a set of nodes (functions state -> partial state update) and edges (static or conditional). After each node returns, the runtime applies the reducer to merge the update into the current state. A compiled graph runs against a thread identified by thread_id; if a checkpointer is attached, every step is persisted as an immutable checkpoint, enabling resume, time-travel, and inspection. interrupt() raises a special exception that surfaces a payload to the caller; the caller resumes by invoking the graph with Command(resume=value).

How would you decide between LangGraph and MAF for a green-field Python-only project?

Both have typed graph workflows, durable checkpoints, time-travel and HITL primitives. Differentiators:

  • If you're Microsoft/Azure-anchored (AAD, App Insights, Foundry, App Service) — pick MAF; declarative YAML agents and Foundry-hosted runtime are real time-savers.
  • If you're already invested in LangChain + LangSmith, or you need the largest open-source community and richest off-the-shelf tools — pick LangGraph.
  • If .NET parity is on the roadmap — MAF is the only credible answer.
  • If cost-of-LLMOps matters and you want a single hosted control plane — LangGraph Platform + LangSmith is the most batteries-included.

7.3 Multi-agent patterns #

What is the Magentic / Magentic-One pattern?

An orchestrator agent maintains a "task ledger" (facts, plan, progress) and a "progress ledger" (open questions, next-action chooser). It dispatches work to specialist agents (web surfer, file surfer, coder, computer terminal). After each step the orchestrator updates the ledger, evaluates whether the plan is still on track, replans if stalled, and decides who acts next. It's effectively a planner-executor with structured self-reflection. Implemented natively in AutoGen (MagenticOneGroupChat) and MAF (Magentic workflow pattern).

How do you prevent a multi-agent system from looping or cost-blowing-up?
  • Hard turn limits (MaxMessageTermination, recursion limit in LangGraph).
  • Budget-per-thread guardrails (token + USD).
  • Termination conditions on agent output (TextMentionTermination("DONE"), structured "done" flags).
  • Cycle detection: if state hasn't changed N steps, escalate to human.
  • Orchestrator-level progress ledger (Magentic) that detects no-progress and replans.
  • Trace + alert on outlier traces in LangSmith / OTel.
Supervisor vs swarm vs hierarchical — when does each shine?

Supervisor (single coordinator routes to workers) is best when one agent should own context and synthesise — e.g. research writer, customer-support orchestrator. Predictable but bottlenecked by the supervisor.
Swarm (peer agents handoff to each other) is best for stateful triage where the "right" specialist owns the conversation and may bounce again — e.g. multi-domain support.
Hierarchical (teams of teams) is best for large bounded problems with clear sub-domains — e.g. MetaGPT's PM/Architect/Engineer/QA.

7.4 Tools & memory #

How are function tools converted to LLM tool schemas under the hood?

Most frameworks introspect the Python signature (or .NET attributes) and a docstring. Type hints + Annotated/Description metadata drive a JSON Schema; the docstring becomes the tool description. The framework registers the schema with the model client (OpenAI tools, Anthropic tools, Ollama function calling). When the LLM emits a tool call, the framework parses arguments against the schema, runs the function, and feeds back a structured tool message. MAF uses AIFunction; AutoGen uses FunctionTool; OpenAI SDK uses @function_tool; Pydantic AI uses Pydantic models for both arguments and outputs.

What memory layers should a production agent have?
  1. Working / context window — current turn + recent N messages.
  2. Thread / session memory — full conversation, summarised when stale.
  3. Episodic memory — distilled facts from prior threads (user preferences, prior decisions).
  4. Semantic / knowledge memory — RAG over docs / KB.
  5. Procedural memory — learned skills (cached plans, code snippets).

MAF expresses (1)–(2) via AgentThread, (3)–(4) via IMemory providers and skills, (5) via tools/sub-workflows. Letta specialises in (3). LangMem/mem0 cover (3)–(4). Avoid pushing everything into the context window — keep it minimal and recall on demand.

How would you implement HITL for a high-risk action like a wire transfer?

Model the workflow as a graph with an explicit approval node. In MAF, use RequestInfoExecutor to pause and emit a structured RequestInfo with the proposed payload; persist a checkpoint; surface to the operator UI; on response, resume. Pair with: tool-call approval middleware (no tool runs without operator OK on a flagged tool), guardrails on input (PII / amount thresholds), audit log written to immutable storage, idempotency keys, and a kill switch via feature flag. Equivalent in LangGraph: interrupt() + checkpointer + Studio approval UI.

7.5 Production concerns #

How do you observe an agent system end-to-end?

Use OpenTelemetry semantic conventions for GenAI:

  • Spans: agent.run, workflow.executor, llm.call, tool.call, handoff, guardrail, request_info.
  • Attributes: model, prompt & completion tokens, cost, tool name, args (redacted), thread_id, user_id, agent_name, parent span.
  • Events: tool_arguments, tool_result, error.
  • Metrics: tokens-per-thread, $-per-thread, tool-error-rate, agent-completion-rate.

Export to Application Insights, Jaeger, Honeycomb, Phoenix, Langfuse, or LangSmith. MAF emits this natively; OpenAI Agents SDK does too.

How do you evaluate an agent?

Three layers:

  1. Component — unit test each tool, prompt, retriever (golden answers, exact-match, schema validity).
  2. Trajectory — judge whether the agent's plan was correct, tools were used appropriately, no hallucinated calls. LLM-as-judge with rubric, or programmatic checks ("did it call X with Y?").
  3. End-to-end — golden-set scoring: correctness, faithfulness, latency, cost, refusal-rate. Run on every PR via LangSmith / Foundry Eval / agent-framework-lab.

Use offline evals for regressions and online evals (sampled production traces graded by an LLM or human) for drift.

How do you secure an agent that calls tools and external APIs?
  • Identity: per-user or per-tenant scoped credentials; never share a global token.
  • Authorization: tool-level RBAC; each tool checks the caller's permissions before executing.
  • Sandboxing: code execution in disposable containers (Docker, ACI) with no host filesystem; egress allow-list.
  • Prompt injection defence: treat tool outputs as untrusted; strip or escape system-prompt-overriding strings; use guardrails to detect injection attempts; consider dual-LLM with a "planner" that never sees raw tool output.
  • Data handling: redact PII in spans; encrypt checkpoints; rotate keys.
  • Rate-limit + budget: per-user token / cost ceilings; circuit breaker.
  • HITL for irreversible actions.
What's the difference between a stateless and a durable agent runtime, and when do you need durable?

A stateless runtime keeps thread state in memory; a crash loses it. A durable runtime persists checkpoints after each step (MAF, LangGraph, Temporal-backed agents) so a crashed or paused workflow can resume exactly where it left off. You need durable when: workflows are long-running (hours/days), HITL pauses span human time, you need to rewind/time-travel for debugging, or you must guarantee at-most-once side effects across retries. For sub-second chat, stateless + idempotent tools is fine.

7.6 MAF deep-dive questions #

What are the core building blocks of MAF?

AIAgent (provider-agnostic chat agent), tools (functions / MCP / OpenAPI / Foundry-hosted), middleware (request/response/exception pipelines), AgentThread (per-conversation state), IMemory providers, Skills, and the Workflow graph engine (Workflow, Executor, Edge, WorkflowBuilder). On top: hosting (local, ASP.NET Core, FastAPI, Foundry-hosted, Agent Harness), DevUI debugger, and front-end adapters (AG-UI, CopilotKit, ChatKit).

What MAF features are GA vs preview as of 1.0?

GA / LTS: agents, workflows, memory, middleware, orchestration patterns (sequential, concurrent, handoff, group chat, Magentic), checkpointing, observability via OTel, declarative YAML agents.

Preview: DevUI, Foundry-hosted agents (turnkey hosting), Foundry-backed memory and tools, evaluations, front-end adapters (AG-UI, CopilotKit, ChatKit), reusable skills, GitHub Copilot SDK and Claude Code SDK integration, and Agent Harness for shell/file/messaging loops.

How do you implement a sequential workflow in MAF (Python)? 
from agent_framework import WorkflowBuilder

workflow = (
    WorkflowBuilder()
        .add_executor(researcher)
        .add_executor(writer)
        .add_executor(editor)
        .add_edge(researcher, writer)
        .add_edge(writer, editor)
        .build()
)

async for event in workflow.run("Write a blog post about MCP"):
    print(event)

Each executor is typed; checkpointer can be attached at build time; RequestInfoExecutor can be inserted anywhere to pause for HITL.

How does MAF stay provider-agnostic?

Through Microsoft.Extensions.AI.IChatClient (and the Python equivalent ChatClient protocol). Concrete implementations exist for Foundry, Azure OpenAI, OpenAI, Anthropic, AWS Bedrock, Ollama, GitHub Copilot SDK and Claude Code SDK. AIAgent takes any IChatClient, so swapping providers is a configuration change. Tools are described via AIFunction/Python annotations and translated to the model-specific tool schema by the chat client.

How would you migrate an AutoGen 0.4 group chat to MAF?
  1. Map AutoGen AssistantAgent → MAF AIAgent with the same tools and instructions.
  2. Replace SelectorGroupChat with a MAF group-chat workflow pattern, or design a typed handoff graph for stricter contracts.
  3. Replace UserProxyAgent with a RequestInfoExecutor node that emits structured RequestInfo events; bind a UI adapter (CopilotKit or AG-UI) to fulfil them.
  4. Wire memory: AutoGen's memory provider → MAF IMemory.
  5. Turn on the OpenTelemetry exporter and DevUI to verify behaviour parity.
  6. Add a checkpointer for durability.

Microsoft publishes a step-by-step migration guide; expect a behavioural-parity period where you run both side-by-side and diff traces.

7.7 System-design exercises #

Design a customer-support agent that handles refunds, can escalate to humans, and is auditable.

Topology. Triage agent (intent classification) → handoff to: (a) FAQ agent (RAG), (b) Refund agent, (c) Live-agent escalation. Refund agent calls a preview-refund tool, then a RequestInfoExecutor for human approval if amount > $100, then commit-refund tool.

State. Per-user thread with conversation, ticket id, user tier. Per-workflow checkpoint after each executor.

Tools. lookup_order, preview_refund, commit_refund, create_ticket, kb_search. All idempotent with operation IDs.

Memory. Thread memory; episodic memory of prior tickets; KB via vector store.

Guardrails. Refund amount cap; PII redaction; refusal on out-of-scope.

Observability. OTel spans + replay for any disputed transaction; immutable audit log.

Eval. Golden 200-ticket set graded weekly on resolution rate, escalation rate, refund accuracy. Online sampled eval on production traces.

Implement in MAF for .NET-shop or LangGraph for OSS-only.

Design a research assistant that produces a 10-page report from a single prompt.

Magentic-style orchestration. Orchestrator builds a task ledger (sub-questions, sources, draft sections); dispatches: Web search agent (Bing tool / Tavily), Reader agent (URL → summary), Synthesiser (per section), Critic (factual, completeness, novelty), Writer (final assembly with citations). Use parallel fan-out for sub-question search, then fan-in for synthesis. Persist intermediate state for resume on failure. Add a HITL approval before writer for cost-sensitive runs. Use LlamaIndex AgentWorkflow for the RAG-heavy parts; orchestrate everything from a MAF or LangGraph workflow.

How would you evaluate three candidate agent frameworks for a 1-year enterprise commitment?

Score each on: licence + redistributability, language parity with your stack, depth of multi-agent primitives, durability/checkpointing, HITL primitives, observability standardisation (OTel), eval tooling, deployment options (managed + self-hosted), security posture (isolation, secrets), governance (RBAC, audit), declarative authoring, vendor lock-in (provider-agnostic LLM clients), community + commercial support, and a 6-month roadmap. Then run the same realistic capstone on each (e.g. multi-agent ticketing assistant) and compare LOC, traces, time-to-first-success, cost per run, and operator UX. Document the verdict with a 2x2 (must-have / nice-to-have).

Behavioural: tell me about a time you had to choose between AutoGen and MAF (or LangGraph and MAF).

Use STAR. Example: "Situation — we needed an internal compliance reviewer that pauses for officer approval. Task — pick a framework. Action — I prototyped both AutoGen 0.4 (familiar to the team) and MAF Python in two days. AutoGen needed a custom checkpoint layer for HITL; MAF's RequestInfoExecutor + checkpointer worked out of the box, and DevUI gave reviewers a live graph view that reduced bug triage time from hours to minutes. Result — chose MAF, shipped in 4 weeks, replay-from-checkpoint cut MTTR by 60%." Pivot to: trade-offs honestly named (smaller community, preview features still moving), and how you mitigated (pinning versions, abstracting the chat client behind an interface).
